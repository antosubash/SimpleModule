# Architecture Patterns: Release Hardening

**Domain:** .NET modular monolith framework hardening
**Researched:** 2026-03-18

## Current Architecture (No Changes to Core)

The existing SimpleModule architecture is sound: source-generator-driven module discovery, per-module DbContext isolation, contract-based inter-module communication, and in-memory event bus. Hardening does not change the architecture -- it fixes gaps in how the architecture is applied.

## Patterns to Follow

### Pattern 1: Fallback Authorization Policy (Deny by Default)

**What:** Configure ASP.NET Core to require authentication on ALL endpoints by default. Public endpoints explicitly opt out with `[AllowAnonymous]`.

**When:** Always. This is the correct default for any application with authentication.

**Why it matters for SimpleModule:** The current issue is that `[RequirePermission]` does not imply `[Authorize]`, so an endpoint can have permission requirements but still be accessible to anonymous users. A fallback policy eliminates this entire class of bug.

**Implementation:**
```csharp
// In Program.cs or module service registration
builder.Services.AddAuthorizationBuilder()
    .SetFallbackPolicy(new AuthorizationPolicyBuilder()
        .RequireAuthenticatedUser()
        .Build());
```

**Affected endpoints that need `[AllowAnonymous]`:**
- Login/register endpoints
- Public landing page
- Health check endpoints
- Static file serving (handled by middleware order, not attributes)

### Pattern 2: Per-Module EF Core Migrations

**What:** Each module maintains its own migration history in its own schema. The host project coordinates migration application at startup (dev) or via CLI (production).

**When:** Any modular monolith with a shared database and per-module schemas.

**Component structure:**
```
modules/Products/src/Products/
  Data/
    ProductsDbContext.cs          # HasDefaultSchema("products")
  Migrations/
    20260318_Initial.cs           # Generated by dotnet ef migrations add
    ProductsDbContextModelSnapshot.cs

modules/Users/src/Users/
  Data/
    UsersDbContext.cs             # HasDefaultSchema("users")
  Migrations/
    20260318_Initial.cs
    UsersDbContextModelSnapshot.cs
```

**Key configuration per module DbContext:**
```csharp
protected override void OnConfiguring(DbContextOptionsBuilder options)
{
    options.UseSqlite(connectionString, o =>
        o.MigrationsHistoryTable("__EFMigrationsHistory", "products"));
}
```

**Migration commands (per module):**
```bash
dotnet ef migrations add Initial \
  --project modules/Products/src/Products \
  --startup-project template/SimpleModule.Host \
  --context ProductsDbContext
```

**Startup behavior:**
- Development: `context.Database.MigrateAsync()` per module on startup
- Testing (SQLite in-memory): Keep `EnsureCreatedAsync()` (migrations don't apply to in-memory)
- Production: `dotnet ef database update` via deployment script

### Pattern 3: EventBus Error Handling Policy

**What:** Configurable per-event execution policy that determines how handler failures are treated.

**When:** Any event bus where handlers have varying reliability requirements.

**Policies:**
```
BestEffort  -- Execute all handlers; collect failures; return aggregate result
FailFast    -- Stop on first failure; throw immediately
```

**Architecture:**
```csharp
public interface IEventBus
{
    Task PublishAsync<T>(T @event, CancellationToken ct = default)
        where T : IEvent;

    Task<EventResult> PublishAsync<T>(T @event, EventPolicy policy, CancellationToken ct = default)
        where T : IEvent;
}

public record EventResult(bool Success, IReadOnlyList<Exception> Failures);
```

**Default:** BestEffort (backward compatible with current behavior, but returns structured result instead of throwing AggregateException).

### Pattern 4: Testcontainers + Respawn Integration Test Fixture

**What:** Single PostgreSQL container per test collection, with Respawn resetting data between tests.

**When:** Integration tests that need production-parity database behavior.

**Architecture:**
```
PostgreSqlFixture : IAsyncLifetime
  - StartAsync: Start Testcontainer, run migrations
  - StopAsync: Dispose container

SimpleModuleWebApplicationFactory (existing, modified)
  - Override connection string to point at Testcontainer
  - Keep SQLite path for unit tests (fast, no Docker needed)

[Collection("PostgreSQL")]
public class ProductsIntegrationTests : IAsyncLifetime
  - InitializeAsync: Respawn.ResetAsync()
  - DisposeAsync: noop
```

**Key principle:** Unit tests stay on SQLite in-memory (fast, no Docker). Integration tests targeting database-specific behavior use Testcontainers (accurate, slightly slower startup).

### Pattern 5: Rate Limiting with Named Policies

**What:** Define rate limit policies by name, attach to specific endpoints.

**When:** Endpoints with abuse potential or compliance requirements.

**Architecture:**
```csharp
// Service registration (centralized)
builder.Services.AddRateLimiter(options =>
{
    options.AddFixedWindowLimiter("personal-data-export", opt =>
    {
        opt.PermitLimit = 1;
        opt.Window = TimeSpan.FromHours(1);
        opt.QueueLimit = 0;
    });
    options.RejectionStatusCode = StatusCodes.Status429TooManyRequests;
});

// Middleware (before authorization but after routing)
app.UseRateLimiter();

// Endpoint (explicit opt-in)
app.MapGet("/users/download-personal-data", handler)
    .RequireRateLimiting("personal-data-export");
```

## Anti-Patterns to Avoid

### Anti-Pattern 1: Global Rate Limiting on All Endpoints
**What:** Applying rate limiting globally via `app.UseRateLimiter()` with a global policy.
**Why bad:** Most endpoints don't need rate limiting. Global limits interfere with legitimate traffic patterns and make debugging harder.
**Instead:** Named policies on specific endpoints that need protection.

### Anti-Pattern 2: Replacing EnsureCreated with Migrate Everywhere
**What:** Calling `Database.MigrateAsync()` in all environments including tests.
**Why bad:** SQLite in-memory databases don't support migrations well. Migration files add noise to test projects.
**Instead:** `MigrateAsync()` for dev/prod, `EnsureCreatedAsync()` for test-only SQLite in-memory.

### Anti-Pattern 3: Admin Permission Check as Boolean Flag
**What:** `if (user.IsAdmin) return true;` bypassing all permission checks.
**Why bad:** No audit trail, no granularity, no way to restrict admin actions.
**Instead:** Admin role gets all permissions assigned explicitly (or via a permission set), so the normal permission check path handles admins too. Admin actions can then be individually revoked if needed.

### Anti-Pattern 4: AggregateException from Event Handlers
**What:** Collecting all handler exceptions and throwing AggregateException.
**Why bad:** Callers must unwrap AggregateException to find individual failures. Partial success is indistinguishable from total failure. Stack traces are confusing.
**Instead:** Return a result object with success/failure per handler. Log failures internally. Let caller decide what to do with the result.

## Component Boundaries (Hardening Additions)

| Component | Responsibility | Communicates With |
|-----------|---------------|-------------------|
| Rate Limiter Middleware | Enforce per-endpoint request limits | ASP.NET pipeline (before authorization) |
| Fallback Authorization Policy | Deny unauthenticated access by default | All endpoints (via policy) |
| Per-Module Migration Runner | Apply schema changes per module DbContext | Module DbContexts, Host startup |
| PostgreSqlFixture (test) | Manage Testcontainer lifecycle | Test collections, WebApplicationFactory |
| Respawn Checkpoint (test) | Reset database state between tests | PostgreSqlFixture |
